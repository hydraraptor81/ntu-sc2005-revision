# Operating Systems Concepts: Processes, Threads, and Memory

## 1. Process States and Transitions
* **Ready vs. Waiting State:** A process waiting to get access to the CPU is in
  the **Ready** state, residing in the ready queue. The **Waiting** (or
  Blocked) state is strictly for processes that cannot execute until a specific
  event occurs, such as the completion of an I/O operation.
* **Hardware Interrupts:** When a hardware interrupt occurs, a process
  currently in the **Running** state is typically preempted and moved to the
  **Ready** state. This allows the CPU to execute the Interrupt Service Routine
  (ISR). Once the ISR finishes, the OS scheduler selects a process from the
  ready queue to run next.
* **The `wait()` System Call:** This system call is used by a **parent
  process** to pause execution until its **child process** completes, not the
  other way around. This is central to the "fork-join" processing model, where
  a parent forks multiple independent child tasks and then joins their results
  upon completion.

## 2. Inter-Process Communication (IPC) Memory Usage
* **Message Passing vs. Shared Memory:** Message passing actually consumes
  **more** memory than shared memory.
* **Why?** In message passing, data must be copied from the sender's user space
  into kernel space, and then copied again from the kernel space into the
  receiver's user space. Shared memory maps a single region of memory directly
  into the address spaces of both processes, avoiding these redundant copies
  and kernel overhead.

## 3. Process Memory: Data Region vs. Stack Region
* **Size Allocation:** The Data region's size is statically fixed at load time.
  In contrast, the Stack region is dynamic; it grows and shrinks automatically
  as the process executes and calls functions.
* **Content:** The Data region is reserved for storing global and static
  variables that persist for the lifetime of the program. The Stack is used for
  temporary, local variables and function call parameters that only exist
  within the scope of a specific function execution.

## 4. Single-Threaded vs. Multi-Threaded Processes
* **Resource Sharing:** In a multi-threaded process, all threads share the same
  code, data, files, and heap regions of memory.
* **Thread Isolation:** Despite sharing memory, every thread has its own unique
  Stack (for its local variables) and its own set of CPU registers. Each thread
  is managed via a Thread Control Block (TCB).
* **Concurrency Benefits:** Multi-threading drastically increases system
  throughput. If one thread in a process gets blocked (e.g., waiting for I/O),
  the OS can simply switch to another thread within the same process, keeping
  the application responsive and making better use of the CPU.

## 5. Context Switching and Kernel Operations When an OS switches execution
between two processes (e.g., Process P0 and Process P1), a "context switch"
occurs.
* **State Transitions:**
    * An executing process (P1) is interrupted: `Running -> Ready`.
    * A new process (P0) is scheduled: `Ready -> Running`.
    * A running process (P0) makes an I/O system call: `Running -> Waiting`.
* **Kernel Operations (The Context Switch):**
    1.  **Save State:** The OS pauses P1, switches to kernel mode, and saves
    P1's current execution state (registers, program counter) into its Process
    Control Block (PCB1).
    2.  **Reload State:** The OS selects P0, reloads its previously saved state
    from PCB0, switches back to user mode, and resumes P0.
    3.  When P0 later traps into the kernel (e.g., to request I/O), its state
    is saved into PCB0, and the OS can reload P1's state from PCB1 to resume
    it.
* **Step-by-Step Interrupt Flow:**
    1. Hardware interrupt arrives.
    2. CPU pauses the running process, switches hardware to kernel mode.
    3. OS saves the context of the running process to its PCB and updates its
    state to "ready".
    4. OS executes the appropriate ISR.
    5. OS selects the next process from the ready queue, updates its state to
    "running".
    6. OS switches hardware back to user mode, loads the new process's context,
    and execution begins.
