# Operating Systems: Critical Section Tutorial Solutions

## Question 1: User-Level Solution Analysis

Consider the following user-level solution to the critical section problem,
where `flag` and `turn` are shared variables initialized to `false` and `0`,
respectively.

### Process Code:
```c
// Process P0
while(1) {
    flag = true;
    while(turn == 1);
    // critical-section
    turn = 1;
    // remainder-section
}

// Process P1
while(1) {
    turn = 0;
    while(flag && turn == 0);
    // critical-section
    flag = false;
    // remainder-section
}
```

### Requirement Evaluation:

**1. Mutual Exclusion: Not Satisfied**
This solution does not guarantee mutual exclusion. Both processes can enter
the critical section at the same time. Consider the following sequence:
1. P1 executes `turn = 0;`
2. P1 checks the condition `while(flag and turn == 0);` and passes because
   `flag` is initially false.
3. P1 enters the critical section.
4. A context switch occurs from P1 to P0.
5. P0 executes `flag = true;`
6. P0 checks `while(turn == 1);` and passes because `turn` is currently 0.
7. P0 enters the critical section.
Result: P0 and P1 are both in the critical section simultaneously.

**2. Progress: Not Satisfied**
This solution does not guarantee progress. A process can be blocked
indefinitely outside the critical section. Consider this sequence:
1. P0 executes the while loop once and completes its critical section,
   setting `turn = 1`.
2. P0 loops back and executes `flag = true;`
3. P0 checks `while(turn == 1);` and gets stuck looping forever because P1
   has not executed to change the `turn` variable back.
Result: Progress is not satisfied for process P0.

**3. Bounded Waiting: Satisfied**
This solution satisfies bounded waiting because there is a strict limit on
how many times one process can enter the critical section while the other is
waiting:
* If P0 is waiting at `while(turn == 1);`, it will enter the moment P1
  executes `turn = 0;` (the first statement in P1's loop). Thus, P1 can
  enter its critical section 0 times after P0 requests access.
* If P1 is waiting at `while(flag and turn == 0);`, it will enter the
  moment P0 executes `turn = 1;` (after P0's critical section). Thus, P0
  can enter its critical section 1 time after P1 has requested access.

## Question 2: True/False Statements

**a) Mutual exclusion can be achieved by disabling interrupts during the**
**critical section.**
* **Answer:** True.
* **Justification:** By disabling interrupts, no other process, including the
  operating system scheduler, will be able to run. This guarantees that the
  current process has exclusive access to the CPU during its critical
  section.

**b) If a solution to a critical section problem satisfies progress, then**
**it also satisfies bounded waiting.**
* **Answer:** False.
* **Justification:** A solution can always favor a specific process (e.g., P0)
  over another (e.g., P1). This satisfies progress because the system as a
  whole continues to make decisions and enter the critical section, but it
  violates bounded waiting because P1 could potentially starve and wait
  indefinitely.

### Example violating Bounded Waiting but satisfying Progress:
```c
// flag is initialized to false, timeout is a large value

// Process P0
while(flag);
flag = true;
// critical section;
flag = false;

// Process P1
while(flag) { wait(timeout); }
flag = true;
// critical section;
flag = false;
```

## Question 3: TestAndSet Hardware-Level Solution

Consider the following implementation of `TestAndSet()` and its use in a
solution where the shared boolean variable `lock` is initialized to `True`.

### Code:
```c
boolean TestAndSet (boolean *target) {
    boolean rv = *target;
    *target = false;
    return rv;
}

// Process Pi (i = 0 or 1)
while(1) {
    while(TestAndSet(&lock) == false);
    // critical-section
    lock = false;
    // remainder-section
}
```

### Requirement Evaluation:

**1. Mutual Exclusion: Satisfied**
If either P0 or P1 successfully enters the critical section, the `lock`
variable becomes `false` (set by the `TestAndSet` function). The other
process calling `TestAndSet` will receive `false` and continue to loop
indefinitely. Thus, only one process can be in the critical section at a
time.

**2. Progress: Not Satisfied**
After either P0 or P1 executes its critical section, it explicitly sets
`lock = false;` in the remainder section. Since `TestAndSet` requires the
target to be `True` to return `True` (allowing a process to break out of
the while loop), and because `lock` is permanently set to `false`, the
critical section remains permanently locked. Any subsequent process trying
to enter will wait indefinitely.
