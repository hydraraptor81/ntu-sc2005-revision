# Operating Systems: Real-Time Systems & Virtualization Tutorial

## 1. True/False Statements

**a) Real-time CPS applications usually require a response within a very short
time.**
* **Answer:** False.
* **Justification:** "Real-time" does not strictly mean "fast" or completing in
a very short time. It means completing within a *bounded time* in the
worst-case scenario. The response times must be guaranteed to meet pre-defined
deadlines, regardless of how long that deadline actually is.

**b) A guest OS abstracts H/W and provides an interface between H/W and
hypervisor.**
* **Answer:** False.
* **Justification:** This describes a Host OS in a Type-2 virtualization setup.
The Guest OS, on the other hand, interacts with the hypervisor and provides
services to the user applications running on top of it.

**c) Managing the interactions between a guest OS and its applications is one
of the key functions of a hypervisor.**
* **Answer:** False.
* **Justification:** The primary function of a hypervisor is managing the
interactions between the Guest OS and the underlying hardware (via virtual
machine management). The interactions between the Guest OS and its own
applications remain completely independent of the hypervisor.

**d) Under a Type-1 para-virtualized hypervisor, the hypervisor directly
interacts with the H/W and provides a virtual call API to interact with the
guest OS.**
* **Answer:** True.
* **Justification:** In Type-1 (bare-metal) virtualization, the hypervisor sits
directly on the hardware. In a para-virtualized environment, not all hardware
instructions are natively virtualized. Therefore, the Guest OS is explicitly
modified to interact with the hypervisor using a virtual call API.

## 2. Virtualized Environment Resource Management

When tasked with setting up a virtualized environment for diverse workloads
like heavyweight data processing and lightweight web servers, three core
resource management strategies apply:

### 1. Emulation
* **Explanation:** Allows running apps/OSs designed for different hardware
environments via instruction translation.
* **Application:** Useful for running legacy data processing systems on modern
servers.
* **Challenge:** Introduces significant performance overhead, slowing down
time-sensitive workloads.

### 2. Multiplexing
* **Explanation:** Sharing physical resources (CPU cores/memory) across multiple
virtual machines by allocating them in time slices.
* **Application:** Highly recommended for **lightweight workloads** like web
servers that do not require continuous CPU access.
* **Challenge:** Poor scheduling or overcommitting can lead to bottlenecks,
especially if heavy workloads get starved.

### 3. Aggregation
* **Explanation:** Pooling multiple physical resources (multiple CPU cores or
servers) into one massive virtual resource.
* **Application:** Highly recommended for **heavyweight workloads** like data
processing that demand more compute power than a single node provides.
* **Challenge:** Reduces resource availability for smaller workloads and adds
management complexity to ensure proper isolation.

**Recommendation Summary:** Prioritize multiplexing for the web servers and
aggregation for the data processing workloads to balance flexibility and
utilization. Emulation should only be used as a last resort for legacy support.

## 3. Earliest Deadline First (EDF) Scheduling

**Process Set Definition (Format: P<Period/Deadline, Execution Time, Deadline>):**
* **P1:** Period = 5, Execution = 2, Deadline = 5
* **P2:** Period = 10, Execution = 3, Deadline = 10
* **P3:** Period = 20, Execution = 5, Deadline = 20

### Schedulability Analysis
The process set **is schedulable** under EDF. All deadlines are met across a
20-unit hyper-period. However, because deadlines coincide at specific intervals,
there are multiple valid schedules based on tie-breaking decisions.

### Gantt Charts: The 4 Possible Schedules
*Note: Timestamps indicate the start/end of process execution blocks.*

**Schedule 1:**
*Tie-breaks won by: P2 over P3 (at t=12), P1 over P3 (at t=15).*
| Time  | 0-2 | 2-5 | 5-7 | 7-10 | 10-12 | 12-15 | 15-17 | 17-19 | 19-20 |
|-------|-----|-----|-----|------|-------|-------|-------|-------|-------|
| **Task**| P1  | P2  | P1  | P3   | P1    | P2    | P1    | P3    | IDLE  |

**Schedule 2:**
*Tie-breaks won by: P2 over P3 (at t=12), P3 over P1 (at t=15).*
| Time  | 0-2 | 2-5 | 5-7 | 7-10 | 10-12 | 12-15 | 15-17 | 17-19 | 19-20 |
|-------|-----|-----|-----|------|-------|-------|-------|-------|-------|
| **Task**| P1  | P2  | P1  | P3   | P1    | P2    | P3    | P1    | IDLE  |

**Schedule 3:**
*Tie-breaks won by: P3 over P2 (at t=12), P1 over P2 (at t=15).*
| Time  | 0-2 | 2-5 | 5-7 | 7-10 | 10-12 | 12-14 | 14-15 | 15-17 | 17-19 | 19-20 |
|-------|-----|-----|-----|------|-------|-------|-------|-------|-------|-------|
| **Task**| P1  | P2  | P1  | P3   | P1    | P3    | P2    | P1    | P2    | IDLE  |

**Schedule 4:**
*Tie-breaks won by: P3 over P2 (at t=12), P2 over P1 (at t=15).*
| Time  | 0-2 | 2-5 | 5-7 | 7-10 | 10-12 | 12-14 | 14-17 | 17-19 | 19-20 |
|-------|-----|-----|-----|------|-------|-------|-------|-------|-------|
| **Task**| P1  | P2  | P1  | P3   | P1    | P3    | P2    | P1    | IDLE  |

### Explanation of Tie-Breaks:
1.  **At t = 12:** P1 finishes its execution. P2 (deadline 20) and P3
    (deadline 20) are both ready. Since their deadlines are identical, the
    scheduler must tie-break (leading to the divergence between Schedules 1/2
    and Schedules 3/4).
2.  **At t = 15:** P1 arrives (deadline 20). Depending on the previous tie-break,
    either P3 (deadline 20) or P2 (deadline 20) still has remaining execution
    time. This forces a second tie-break, splitting the paths again into the 4
    final schedules shown above.
