# Operating Systems: File Systems Tutorial Solutions

## Question 1: File Deletion and Sharing

### (a) Phantom Files and Deletion
**Problem:** If a file is deleted and its space reclaimed while hard links
still exist, a new file might be created in that exact storage area. Accessing
the old link results in a "phantom" file scenario.
* **Data Content Issue:** The user accidentally accesses the new file's data.
* **Access Protection Issue:** The new file is accessed without checking its
  specific access permissions (since the old link's permissions are used).

**Solution:** * Ensure all links are deleted when the file is deleted.
* Alternatively, only allow file deletion after all links have been removed.

### (b) File Sharing: Single Copy vs. Multiple Copies
* **Single Copy Approach:**
  * *Merit:* Saves storage space and ensures data consistency across all users.
  * *Demerit:* Concurrent updates cause race conditions; requires strict mutual
    exclusion.
* **Multiple Copies Approach:**
  * *Merit:* Better performance as mutual exclusion is not necessary.
  * *Demerit:* Wastes storage space and copies can easily become inconsistent.

## Question 2: Unix-like Inodes and Disk Operations

**System Specs:** Unix-like inode (6 direct, 1 single-indirect pointer). Block
size is 1000 bytes.
**Program Code:**
`fd = open("/usr/ast/mbox");`
`seek(fd, 5900);`
`read(fd, buf, 200);`

### (a) Disk Reads for `open`
Assuming the root directory `/` is in memory and inodes are on disk:
1. Load inode for `usr`.
2. Load data block for `usr`.
3. Load inode for `ast`.
4. Load data block for `ast`.
5. Load inode for `mbox`.
**Total:** 5 disk read operations.

### (b) Disk Reads for `seek` and `read`
* **`seek(fd, 5900)`:** Requires **0** disk reads. It simply updates the
  current position pointer in kernel memory.
* **`read(fd, buf, 200)`:** Reads 200 bytes starting from byte 5900 (up to
  byte 6099).
  * Byte 5900 falls in logical block 5 (bytes 5000-5999).
  * Byte 6000 falls in logical block 6 (bytes 6000-6999).
  * Block 5 uses a direct pointer (reads data immediately).
  * Block 6 uses the single-indirect pointer (must read index block first).
  * **Operations:** Read block 5 + Read indirect block + Read block 6.
  * **Total:** 3 disk read operations.

### (c) Maximum File Size
Each indirect block entry takes 2 bytes.
* Direct pointers = 6
* Indirect pointers per block = 1000 bytes / 2 bytes = 500 pointers.
* Max file size = (6 direct + 500 indirect) * 1000 bytes/block = **506,000 bytes**

## Question 3: File Allocation Methods and I/O Count

**Scenario:** Read 200 bytes starting from byte 1900 (bytes 1900-2099).
Block size = 1000 bytes. This targets logical blocks 1 and 2. The directory
entry is already cached in memory.

| Allocation Method | Disk I/O Count | Explanation |
| :--- | :--- | :--- |
| **Contiguous** | 2 | Directly read block 1 and block 2. |
| **Linked** | 3 | Must read block 0 to find the pointer to block 1, then read block 1, then block 2. |
| **Indexed** | 3 | Read the index block to find pointers, then read block 1, then block 2. |

## Question 4: Mixed Block Sizes

Some file systems allocate storage using mixed block sizes (e.g., 4-Kbyte
large blocks and 512-byte small blocks).

**Advantage:** This hybrid approach combines the benefits of both sizes:
* **Small blocks** reduce internal fragmentation (less wasted space for small
  files or the tail ends of large files).
* **Large blocks** improve throughput (faster read/write speeds for bulk data
  transfer).
