# Operating Systems: Memory Management Tutorial Solutions

## Question 1: Memory Allocation & Compaction

A system with 4800 words of memory supports multiple partitions.
Initial Occupied Memory:
* Address 1000, Length 1200
* Address 3100, Length 700
* Address 4000, Length 400
* Address 4400, Length 200

**Rules for New Requests:**
1. Use allocation algorithm (Best-fit or First-fit).
2. If no hole is large enough, perform memory compaction by moving occupied
   partitions toward lower addresses.
3. Stop compaction the moment a sufficiently large hole is formed.

### a) Best-Fit Allocation Trace
Requests: 600, 250, and 1050 words.

1.  **Request 600:** Best fit is the 900-word hole at address 2200.
    * Allocated at: `2200 - 2800`
    * Remaining hole: `2800 - 3100` (Size: 300)
2.  **Request 250:** Best fit is the 300-word hole at address 2800.
    * Allocated at: `2800 - 3050`
    * Remaining hole: `3050 - 3100` (Size: 50)
3.  **Request 1050:** No hole is large enough. **Compaction begins:**
    * Move block 1200 (at 1000) to 0. (Hole at 1200 is size 1000 - too small)
    * Move block 600 (at 2200) to 1200.
    * Move block 250 (at 2800) to 1800.
    * *Resulting Hole:* From 2050 to 3100 (Size: 1050). Compaction stops!
    * Allocate 1050 at: `2050 - 3100`

**Final Memory State (Best-Fit):**
| Start Addr | End Addr | Status             | Size  |
| :---       | :---     | :---               | :---  |
| 0          | 1200     | Occupied (Moved)   | 1200  |
| 1200       | 1800     | Occupied (Req 600) | 600   |
| 1800       | 2050     | Occupied (Req 250) | 250   |
| 2050       | 3100     | Occupied (Req 1050)| 1050  |
| 3100       | 3800     | Occupied           | 700   |
| 3800       | 4000     | Free Hole          | 200   |
| 4000       | 4400     | Occupied           | 400   |
| 4400       | 4600     | Occupied           | 200   |
| 4600       | 4800     | Free Hole          | 200   |

---

### b) First-Fit Allocation Trace
Requests: 600, 250, and 1050 words.

1.  **Request 600:** First fit is the 1000-word hole at address 0.
    * Allocated at: `0 - 600`
    * Remaining hole: `600 - 1000` (Size: 400)
2.  **Request 250:** First fit is the 400-word hole at address 600.
    * Allocated at: `600 - 850`
    * Remaining hole: `850 - 1000` (Size: 150)
3.  **Request 1050:** No hole is large enough. **Compaction begins:**
    * Move block 1200 (at 1000) down to 850.
    * *Resulting Hole:* From 2050 to 3100 (Size: 150 + 900 = 1050).
    * Compaction stops! Allocate 1050 at: `2050 - 3100`

**Final Memory State (First-Fit):**
| Start Addr | End Addr | Status             | Size  |
| :---       | :---     | :---               | :---  |
| 0          | 600      | Occupied (Req 600) | 600   |
| 600        | 850      | Occupied (Req 250) | 250   |
| 850        | 2050     | Occupied (Moved)   | 1200  |
| 2050       | 3100     | Occupied (Req 1050)| 1050  |
| 3100       | 3800     | Occupied           | 700   |
| ...        | ...      | (Identical to Best-Fit after 3100) |

---

### c) Relocation Overhead Comparison
* **Best-Fit:** Moved blocks of 1200 + 600 + 250 = **2050 words**.
* **First-Fit:** Moved block of **1200 words**.
* **Conclusion:** In this scenario, **First-Fit yields lower overhead**.

## Question 2: Fragmentation & Absolute Addresses

**a) Schemes suffering from internal fragmentation:**
* Fixed partitioning
* Paging

**b) Why memory compaction fails with absolute address format:**
If absolute addressing is used, the memory addresses for data and code are
hardcoded (bound to physical hardware addresses) at compile time. Compaction
requires moving this code/data to new physical locations. Moving them would
invalidate the hardcoded addresses, resulting in addressing errors.

## Question 3: Memory Access Time (Paging)

Given a memory reference time of 200 nanoseconds (ns):

**a) Paged memory reference time:**
* Requires 2 memory accesses:
    1. Access the page table in memory (200ns).
    2. Access the actual physical word in memory (200ns).
* **Total Time:** 200ns + 200ns = **400ns**.

**b) Effective Access Time (EAT) with Associative Registers (TLB):**
* TLB Hit Rate: 75% (0.75)
* TLB Hit Access Time: 0ns (TLB search) + 200ns (memory word) = 200ns.
* TLB Miss Access Time: 400ns (page table + memory word).
* **Calculation:** EAT = (0.75 * 200ns) + (0.25 * 400ns)
    EAT = 150ns + 100ns = **250ns**.

## Question 4: Address Formatting

**System Specs:**
* Logical Address: 32 bits total.
* Page Size: 1 KB (1024 bytes = 2^10 bytes).
* Physical Memory: 1 GB (2^30 bytes).

**a) Format of Logical and Physical Addresses:**
* **Offset:** log2(Page Size) = log2(2^10) = **10 bits** (same for both).
* **Logical Address (32 bits):** * Page Number = Total Bits - Offset = 32 - 10 = **22 bits**.
    * Format: `[Page Number: 22 bits] | [Offset: 10 bits]`
* **Physical Address (30 bits):** * Frame Number = Total Bits - Offset = 30 - 10 = **20 bits**.
    * Format: `[Frame Number: 20 bits] | [Offset: 10 bits]`

**b) Entries in a Standard Page Table:**
A standard page table requires an entry for every possible logical page.
* Entries = 2^(Page Number Bits) = **2^22 entries**.

**c) Entries in an Inverted Page Table:**
An inverted page table has exactly one entry for every physical frame.
* Entries = 2^(Frame Number Bits) = **2^20 entries**.
