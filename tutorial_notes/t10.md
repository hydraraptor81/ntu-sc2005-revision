## Question 1: LRU Page Replacement & Reference String Reverse-Engineering

A system uses demand paging with 4 page frames and the Least Recently Used (LRU)
page replacement algorithm. The process requires pages 1, 2, 3, 4, and 5.
Given the sequence of **paged out** pages: `3, 1, 2, 4, 5, N, 1`
*(where 'N' means no page was paged out).*

**a) What are the pages initially loaded in the frames?**
* **Answer:** 3, 1, 2, 4.
* **Logic:** The first four page faults evict pages 3, 1, 2, and 4 in that
  exact order. For an LRU system, this means these pages were already in memory
  and were the least recently used at the time of their respective evictions.

**b) What is the corresponding page reference string?**
* **Answer:** `5, 3, 1, 2, 4, 3, 5`
* **Explanation via LRU Stack Implementation:**
  *(Note: Top of stack is Most Recently Used; Bottom is Least Recently Used)*

| Ref | Paged Out | Stack State (Top to Bottom) | Explanation |
| :-- | :-------- | :-------------------------- | :--------------------------- |
| Initial| -      | [4, 2, 1, 3]                | Based on the eviction order. |
| **5** | 3         | [5, 4, 2, 1]                | 5 loaded, LRU page 3 evicted.|
| **3** | 1         | [3, 5, 4, 2]                | 3 loaded, LRU page 1 evicted.|
| **1** | 2         | [1, 3, 5, 4]                | 1 loaded, LRU page 2 evicted.|
| **2** | 4         | [2, 1, 3, 5]                | 2 loaded, LRU page 4 evicted.|
| **4** | 5         | [4, 2, 1, 3]                | 4 loaded, LRU page 5 evicted.|
| **3** | N         | [3, 4, 2, 1]                | 3 is referenced (page hit!). |
| **5** | 1         | [5, 3, 4, 2]                | 5 loaded, LRU page 1 evicted.|

## Question 2: Fixed Allocation (LRU) vs. Working Set Model

A process is allocated 4 page frames. The initial state is given below:

| Frame # | Page # | Time Loaded | Time Last Accessed |
| :------ | :----- | :---------- | :----------------- |
| 0       | 1      | 60          | 161                |
| 1       | 0      | 130         | 160                |
| 2       | 3      | 26          | 162                |
| 3       | 2      | 20          | 163                |

*Initial LRU Order (MRU to LRU):* 2, 3, 1, 0.
*Reference String:* `4, 0, 0, 0, 2, 4, 2, 1, 0, 3, 2`

**a) Page Faults using Fixed Allocation LRU (4 Frames)**
Consecutive identical references (like 0, 0, 0) do not change the LRU stack top,
so they are condensed in the analysis.

| Ref String | 4 | 0 | 2 | 4 | 2 | 1 | 0 | 3 | 2 |
| :--------- | :-| :-| :-| :-| :-| :-| :-| :-| :-|
| **Stack 1**| 4 | 0 | 2 | 4 | 2 | 1 | 0 | 3 | 2 |
| **Stack 2**| 2 | 4 | 0 | 2 | 4 | 2 | 1 | 0 | 3 |
| **Stack 3**| 3 | 2 | 4 | 0 | 0 | 4 | 2 | 1 | 0 |
| **Stack 4**| 1 | 3 | 3 | 3 | 3 | 0 | 4 | 2 | 1 |
| **Out Page**| 0 | 1 | - | 3 | - | 3 | 3 | 4 | - |
| **Fault?** | P | P | - | P | - | P | - | P | - |

*Wait, examining the slides closely reveals only 4 faults total in the exact
solution trace:*
`4` (fault, out 0), `0` (fault, out 1), `2` (hit), `4` (hit), `2` (hit),
`1` (fault, out 3), `0` (hit), `3` (fault, out 4), `2` (hit).
**Total Page Faults: 4**

**b) Page Faults using Working Set Model (Window Size = 4)**
Instead of fixed frames, only pages within the last 4 references are kept in
memory. Initial working set based on past access times = {0, 1, 2, 3}.

| Ref | Active Window History | Working Set Contents | Fault? |
| :-- | :-------------------- | :------------------- | :----- |
| Initial| 0, 1, 3, 2           | {0, 1, 2, 3}         | -      |
| **4** | 1, 3, 2, 4            | {1, 2, 3, 4}         | **P** |
| **0** | 3, 2, 4, 0            | {0, 2, 3, 4}         | **P** |
| **0** | 2, 4, 0, 0            | {0, 2, 4}            | -      |
| **0** | 4, 0, 0, 0            | {0, 4}               | -      |
| **2** | 0, 0, 0, 2            | {0, 2}               | **P** |
| **4** | 0, 0, 2, 4            | {0, 2, 4}            | **P** |
| **2** | 0, 2, 4, 2            | {0, 2, 4}            | -      |
| **1** | 2, 4, 2, 1            | {1, 2, 4}            | **P** |

## Question 3: System Performance & Thrashing Diagnosis

**System Profile:**
* CPU Utilization: 20%
* Paging Disk Utilization: 97.7%
* Other I/O Devices: 5%

**Diagnosis:** The system is clearly **thrashing**. It is spending nearly all of
its time swapping pages in and out of the disk rather than executing processes.

**Evaluating Solutions to Improve CPU Utilization:**

* **a. Install a faster CPU:** **No.** The CPU is currently starved for data,
  not overloaded. A faster CPU will just idle faster.
* **b. Install a bigger paging disk:** **No.** Capacity is not the issue; the
  transfer speed and memory size are the bottlenecks.
* **c. Increase the degree of multiprogramming:** **No.** Adding more processes
  will increase memory contention and make thrashing worse.
* **d. Decrease the degree of multiprogramming:** **Yes.** This frees up page
  frames for the remaining processes, allowing them to establish their working
  sets and stop thrashing.
* **e. Install more main memory:** **Yes.** This gives all processes enough
  frames to hold their working sets without constantly swapping.
* **f. Install a faster hard disk / multiple disks:** **Probably yes.** By
  removing the disk I/O bottleneck with faster response times, the CPU receives
  data quicker and utilization will increase.
* **g. Increase the page size:** **Depends on memory access pattern.** * If access is highly sequential, larger pages reduce total page faults.
  * If access is highly random, larger pages waste memory by bringing in unused
    data, which worsens paging activity since fewer pages can fit in memory.
