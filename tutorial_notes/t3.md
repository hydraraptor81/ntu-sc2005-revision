# Operating Systems: Scheduling Tutorial Solutions

## Question 1: True/False Statements

* **(a) When a new process is admitted in the system, the short-term scheduler
  must execute in order to keep the CPU busy.**
    * **Answer:** False.
    * **Justification:** The short-term scheduler primarily executes when a
      currently running process terminates or blocks. If a new process is
      admitted, the scheduler only *needs* to execute to keep the CPU busy if
      there are no other processes currently running (i.e., the system is
      idle).
* **(b) Partitioned multi-processor scheduling suffers from migration overheads
  due to data in private core-specific caches.**
    * **Answer:** False.
    * **Justification:** Migration overheads occur in *global* scheduling,
      where a process might execute on one core and then migrate to another,
      invalidating cache states. In partitioned scheduling, processes are
      bound to specific cores and do not migrate. Instead, partitioned
      scheduling suffers from load balancing issues depending on how processes
      are mapped to cores.

## Question 3: Dual-Core Schedule Analysis

**Arrival Times:**
* **Time 0:** P0, P1, P4, P5, P6 (Tie-breaker order: P0 -> P1 -> P4 -> P5 ->
  P6)
* **Time 3:** P2
* **Time 9:** P3

### Visualized Dual-Core Schedule
*Core 1 Timeline:* `[0]-P0-[1]-P4-[4]-P2-[5]-P0-[7]-P4-[9]-P2-[10]-P3-[12]`
*Core 2 Timeline:* `[0]-P5-[2]-P6-[6]-P1-[12]`
*I/O Timeline:* `[1]-P0-[3]` | `[4]-P4-[7]` | `[5]-P2-[7]`

### 3a) Process Burst Lengths (Extracted from Gantt Chart)
* **P0:** 1, 2, 2 (CPU, I/O, CPU)
* **P1:** 6 (CPU only)
* **P2:** 1, 2, 1 (CPU, I/O, CPU)
* **P3:** 2 (CPU only)
* **P4:** 3, 3, 2 (CPU, I/O, CPU)
* **P5:** 2 (CPU only)
* **P6:** 4 (CPU only)

### 3b) Identifying the Scheduling Algorithm
* **Partitioned Scheduling?** **Yes.**
    * P1, P5, and P6 execute exclusively on Core 2. P0, P2, P3, and P4
      execute exclusively on Core 1.
    * *Core 2 algorithm:* Cannot be FCFS or RR (otherwise P1 would run first
      due to tie-breaker). It operates as **Shortest Job First (SJF)** or
      **Shortest Remaining Time First (SRTF)** based on the burst lengths.
    * *Core 1 algorithm:* Cannot be FCFS (P0 would run at time 4 instead of
      P2). Cannot be RR (quantum would need to be >= 3, which breaks FCFS
      order). It must be **SJF** or **SRTF**.
* **Global Scheduling?** **Yes, potentially.**
    * *FCFS or RR:* Impossible, because P1 would have executed in parallel
      with P0 at the very start.
    * *SRTF:* Impossible, because P6 (running on Core 2) would have been
      preempted by the shorter P2 when it arrived at time 3.
    * *SJF:* This is possible if SJF prioritizes strictly based on the current
      burst length. At time 1, P4 (burst 3) is prioritized over P6 (burst 4).

## Question 4: CPU Efficiency Formulas

**Definition:** CPU Efficiency represents the ratio of useful CPU execution time
against the total time the CPU spends (which includes context switch overhead).

**Formula:** `Efficiency = Useful CPU Time / Total CPU Time = T / (T + S * #Ctx_Switch)`

*Note: The first context switch to start the process is also counted as
overhead.*

**Efficiency across different Round-Robin Quantums (Q):**
*(Where T = Average process run time before I/O block, S = Context switch
overhead)*

| Scenario | # Context Switches | CPU Efficiency Formula | Explanation |
| :--- | :--- | :--- | :--- |
| **(a) Q = infinity** | 1 | `T / (S + T)` | The process runs its entire burst without preemption. Only 1 initial context switch. |
| **(b) Q > T** | 1 | `T / (S + T)` | The process finishes its burst before the quantum expires. |
| **(c) S < Q < T** | ceil(T/Q) | `T / (T + S * ceil(T/Q))` | The process is preempted multiple times before finishing its burst. |
| **(d) Q = S** | ceil(T/S) | `T / (T + S * ceil(T/S))` | Quantum equals overhead time. Efficiency is approximately 50% (0.5). |
| **(e) Q = 0** | Infinity | `0` | The CPU spends all its time context switching and performs zero useful work. |
