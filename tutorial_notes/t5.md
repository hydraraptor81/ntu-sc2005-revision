# Operating Systems Tutorial: Synchronization & Semaphores

## 1. Multiplayer Game (Deadlock Avoidance)

Alice needs to pair 1 apple with 2 oranges using semaphores A (apples)
and O (oranges). To avoid deadlock with other players, semaphores must
not be held in a nested manner (holding one while waiting for another).

**Completed Code:**
```c
Wait(A);                      // 1. Acquire Apple basket
// Pick 1 apple
Signal(A);                    // 2. Release Apple basket immediately

Wait(O);                      // 3. Acquire Orange basket
if (atleast 2 oranges in basket) {
    // Pick 2 oranges and pair with apple.
    Signal(O);                // 3. Release Orange basket
} else {
    Signal(O);                // 4. Release Orange basket
    Wait(A);                  // 4. Re-acquire Apple basket
    // Drop 1 apple back in basket
    Signal(A);                // 5. Release Apple basket
}
```

**Justification:**
There is no deadlock because the two semaphores are never acquired
together. By releasing semaphore A before acquiring semaphore O, the
system avoids circular wait conditions (which cause deadlocks).

## 2. Semaphore System Calls Evaluation

Evaluating a modified semaphore where `Wait()` blocks if `S.value < -1`
and `Signal()` wakes up if `S.value <= 0`. Initialized to `S = 0`.

**Mutual Exclusion: Satisfied**
The first process executing `Wait()` decrements `S.value` to `-1`. Since
`-1 < -1` is false, it enters the critical section (CS). If a second
process executes `Wait()`, `S.value` becomes `-2`. Since `-2 < -1` is
true, it blocks. Thus, only one process can enter the CS at a time.
Because `Wait()` and `Signal()` are executed in pairs, the value never
exceeds `0`.

**Progress: Satisfied**
If `S.value` is less than `-1`, processes are blocked. A process exiting
the CS calls `Signal()`, which increments `S` and executes `wakeup()`,
unblocking a process. If `S` increments from `-1` to `0`, `Signal()`
will still execute an unnecessary `wakeup()` (which is harmless). No
process is blocked while the semaphore is available (i.e., value is 0).

## 3. Semaphore Implementation using TestAndSet

Implementing the critical section of the semaphore operations themselves
using a hardware `TestAndSet` instruction and a shared `lock` boolean.

**Wait(S) Implementation:**
```c
// Shared variable: Boolean lock = false;

// Acquire lock for semaphore's critical section
while TestAndSet(&lock);

S.value = S.value - 1;
if (S.value < 0) {
    // Add current process to S.L (wait queue)
    add current process to S.L;

    // Release lock BEFORE going to sleep to avoid deadlock
    lock = false;

    Put current process to waiting state;
} else {
    // Release lock
    lock = false;
}
```

**Signal(S) Implementation:**
```c
// Acquire lock for semaphore's critical section
while TestAndSet(&lock);

S.value = S.value + 1;
if (S.value <= 0) {
    // Remove a process P from S.L (wait queue)
    remove a process P from S.L;

    // Release lock BEFORE waking up the process
    lock = false;

    Add process P to ready queue;
} else {
    // Release lock
    lock = false;
}
```

**Key Takeaway:**
The `TestAndSet` loop protects the manipulation of `S.value` and `S.L`.
Crucially, the lock must be set to `false` right before putting a process
to sleep or waking one up. If a process blocks while holding the `lock`,
no other process will ever be able to signal it, causing a deadlock.
